# 검색 API 부하테스트 — 동시성 병목 2건 제거로 응답시간·처리량 개선

> Spring Boot 구인 검색 API를 k6로 부하테스트(50 VU·60초)해 동시성 병목을 측정하고,
> ① 검색당 중복 집계 쿼리 제거(facet 캐시) ② 커넥션 풀 적정화(Hikari 10→30) 두 단계로
> **검색 중앙 응답시간 −82% · p95 −62% · p99 −37% · 처리량 +22% (에러 0%)** 달성.

## TL;DR — 누적 개선 (검색 API `/api/v1/jobs`, 동일 50 VU·60초)

| 지표 | 최초 | 최종 | 누적 개선 |
|---|---:|---:|---:|
| **중앙값(median)** | 147.9 ms | 27.3 ms | **−81.6%** |
| 평균(avg) | 190.6 ms | 52.5 ms | −72.5% |
| p90 | 364.5 ms | 102.4 ms | −71.9% |
| **p95** | 463.1 ms | 177.9 ms | **−61.6%** |
| **p99** | 768.4 ms | 486.5 ms | **−36.7%** |
| **처리량(throughput)** | 79.3 req/s | 96.9 req/s | **+22.1%** |
| 에러율 | 0.00% | 0.00% | — |

## 측정 방법 (재현 가능)

- **도구**: k6 (Docker `grafana/k6`), 내장 web dashboard로 HTML 리포트 생성.
- **부하**: 50 constant VUs · 60초 · think time 0.2–0.7s · `Accept-Encoding: gzip`.
- **워크로드**: 검색 70%(랜덤 q·region·sort·page) / 회사 목록 12% / 회사별 공고(페이징) 18%.
- **대상**: Spring Boot 백엔드 `:8080` 직접 호출(프론트 우회 → API 성능만 분리).
- **공정성**: 변경 변수 하나만 바꿔 연속 측정, 각 측정 전 백엔드 워밍업(JIT), 동일 로컬 머신.
  머신 노이즈가 절대값을 흔들어, **각 라운드의 비교 기준선을 변경 직전 새로 측정**함.
- **데이터 규모**: 활성 공고 ≈ 5,600건. (Postgres `max_connections`=100.)

## 라운드 1 — 검색당 중복 집계 쿼리 제거 (facet 캐시)

**병목**: 검색 요청마다 필터 집계(facet)를 위해 비자/원격/원격자격 분포를 구하는 `GROUP BY`
3개를 매번 실행 → 검색당 DB 작업 4배(메인 검색 + 집계 3).
**관찰**: 이 집계는 **검색 조건과 무관한 전역 분포**라 모든 검색에 동일, 값은 ETL(일 단위)로만 변함.
**해결**: facet 결과를 **60초 TTL 인메모리 캐시**(`JobService.facets()`)로 보관 → 검색당 집계 3개 제거.
**정확성**: 캐시 전/후 facet 응답 값 **완전 동일**(diff 검증).

| 검색 API | 전(집계 매번) | 후(캐시) | 개선 |
|---|---:|---:|---:|
| median | 147.9 ms | 42.9 ms | −71.0% |
| p90 | 364.5 ms | 182.6 ms | −49.9% |
| p95 | 463.1 ms | 378.1 ms | −18.4% |
| p99 | 768.4 ms | 765.5 ms | −0.4% |
| throughput | 79.3 req/s | 90.9 req/s | +14.5% |

→ **typical 경로(중앙값·p90)는 크게 개선, 꼬리(p95/p99)는 거의 그대로**. 꼬리는 집계가 아니라
커넥션 풀 경합에 묶여 있다는 가설 → 라운드 2.
캡처: `baseline.png`(전) / `after.png`(후).

## 라운드 2 — 커넥션 풀 적정화 (Hikari 10 → 30)

**가설**: 풀 10개로 50 VU를 받으면 커넥션 큐잉이 꼬리 지연을 만든다("중앙값 정상 + 꼬리 폭발" 패턴).
**검증**: 부하 중 DB 커넥션 수 11(풀10) → **31(풀30, 풀 maxed)** 확인.
**변경**: `spring.datasource.hikari.maximum-pool-size: 10 → 30` (`${DB_POOL_SIZE:30}`).

| 검색 API (facet 캐시 ON 고정) | 풀 10 | 풀 30 | 개선 |
|---|---:|---:|---:|
| median | 33.3 ms | 27.3 ms | −18.1% |
| p90 | 197.6 ms | 102.4 ms | −48.2% |
| **p95** | 377.7 ms | 177.9 ms | **−52.9%** |
| **p99** | 684.7 ms | 486.5 ms | **−28.9%** |
| throughput | 91.2 req/s | 96.9 req/s | +6.2% |

→ **꼬리(p95 −53%·p99 −29%)가 크게 개선** — 풀 경합 가설 확인. 라운드 1이 못 줄인 꼬리를 직격.
캡처: `pool10.png`(전) / `pool30.png`(후).

## 분석

- 두 최적화가 **상호 보완**: facet 캐시는 typical 경로(중앙값·평균·p90)를, 풀 적정화는 꼬리(p95·p99)를
  각각 줄여, 합산해 검색 응답시간 분포 전체가 좌측 이동.
- **안정성**: 모든 측정에서 **에러 0%**(읽기 엔드포인트, 레이트리밋 미적용, 5xx 없음).
- 근본 원인을 데이터로 짚고(집계 중복 → 캐시, 풀 경합 → 커넥션 수), 가설→측정→검증 사이클로 진행.

## 한계 (해석 주의)

- **로컬 단일 머신**(백엔드·Postgres·AI 서버·k6 공존)이라 절대 수치는 보수적 → **상대 개선** 중심.
  운영(분리 인스턴스·다중 코어)에선 절대 지연이 더 낮을 것.
- 머신 노이즈로 런마다 절대값 변동 → 각 라운드 비교 기준선을 **변경 직전 새로 측정**해 통제.
- 풀 상향은 DB가 동시 쿼리를 감당할 여력이 있을 때 유효(여기선 `max_connections`=100, CPU 여유 확인).
